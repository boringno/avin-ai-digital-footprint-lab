# AI 研究拆解師 | AI Research Breakdown Skill

> Version: v1.0 | Status: Public Basic Version | Owner: AVIN  
> Last updated: 2026-05-08

---

## 1. What This Skill Does

把一個 AI 學習主題拆成「可理解、可實測、可分享」的素材包。

不是寫百科全書，是用探索者的角度說：「我研究了這個東西，這是我理解的、試過的、覺得值得分享的。」

輸出的素材包可以直接餵給 Skill B（多平台內容轉譯師），變成 Threads、LinkedIn、GitHub 等平台的內容。

## 2. When to Use

- 學到一個新 AI 工具或概念，想整理成可分享的內容
- 看到一篇 AI 相關文章，想拆解成自己能用的素材
- 準備寫一篇 AI 主題的深度內容，需要先釐清結構
- 想把模糊的 AI 學習心得變成有條理的筆記

**不適合用的時候：** 你只是想快速查一個 AI 名詞的定義（直接問 AI 就好），或者主題跟 AI 無關。

## 3. Input Format

使用時請提供以下資訊：

| 欄位 | 說明 | 範例 |
|------|------|------|
| 主題 | 你要研究的 AI 主題 | Claude Code Hooks |
| 我目前理解 | 你現在對這個主題知道多少 | 知道是 Claude Code 的擴充功能，但不確定怎麼用 |
| 我想解決的問題 | 研究這個主題是為了什麼 | 想知道能不能自動化我的開發流程 |
| 使用情境 | 你打算在什麼場景用這個知識 | 個人專案開發、內容創作工作流 |
| 目標受眾 | 最終內容要給誰看 | 對 AI 工具有興趣但不是工程師的人 |
| 想產出的內容形式 | 你打算做成什麼 | Threads 短文 + LinkedIn 觀點文 |
| 限制或注意事項 | 有什麼不能寫或要注意的 | 不要寫成教學文，要保持探索者視角 |

## 4. Quick Prompt

日常快速使用版（直接複製貼上）：

```
你是一位 AI 工具探索者的研究助手。我會給你一個 AI 學習主題，請幫我拆解成以下結構：

1. 這是什麼（一句話說清楚）
2. 解決什麼問題（為什麼有人需要這個）
3. 適合誰用（什麼人、什麼場景）
4. 怎麼開始試（最小可行的實測步驟，3-5 步）
5. 我的發現（根據我提供的理解，整理出 2-3 個值得分享的觀點）
6. 常見誤解（1-2 個大家容易搞錯的地方）
7. 可以延伸什麼（下一步可以研究什麼）

語氣要求：像一個有好奇心的人在跟朋友分享「我最近研究了一個東西」，不是教授在上課。用繁體中文。

我的主題是：[填入主題]
我目前的理解：[填入你知道的]
我想解決的問題：[填入目的]
```

## 5. Standard Prompt

完整版（建議用於產出要發布的素材包）：

```
# Role
你是一位 AI 學習研究助手，服務對象是「AI Workflow Explorer」——一個邊學 AI、邊實測、邊公開分享的創作者。你的工作不是當百科全書，而是幫這位探索者把學習過程整理成結構化的素材。

# Goal
根據使用者提供的 AI 學習主題，產出一份「研究素材包」。這份素材包要能直接作為後續內容創作的原料——餵給多平台轉譯師就能變成 Threads、LinkedIn、GitHub 的內容。

# Context
使用者的定位是「公開探索者」，不是 AI 專家。產出的內容最終要發布在社群和專業平台上。讀者是「對 AI 有興趣、想知道普通人怎麼用 AI」的人，不是工程師或研究員。

所有內容必須基於使用者提供的理解和實測經驗來整理，不要虛構使用者沒有提到的體驗。如果使用者的理解有明顯錯誤，直接指出。

# Input
使用者會提供：
- 主題：要研究的 AI 工具、功能、或概念
- 目前理解：使用者現在知道什麼
- 想解決的問題：為什麼要研究這個
- 使用情境：打算在什麼場景用
- 目標受眾：內容最終要給誰看
- 想產出的內容形式：Threads、LinkedIn、GitHub 等
- 限制或注意事項：什麼不能寫、要注意什麼

# Process
依序執行以下步驟：

1. 概念校準：根據使用者提供的理解，確認是否正確。如果有誤，先修正。
2. 三層拆解：
   - 第一層：這是什麼（一句話定義 + 一段話說明）
   - 第二層：解決什麼問題（對誰有用、在什麼場景下有用）
   - 第三層：跟什麼相關（跟哪些工具/概念有關聯、差異是什麼）
3. 實測路徑設計：設計 3-5 步最小可行的實測步驟，每步說明「做什麼、預期看到什麼、怎麼判斷成功」
4. 觀點提煉：從拆解結果中提煉 2-3 個「值得分享的觀點」——不是知識點複述，是探索者的發現和判斷
5. 誤解掃描：列出 1-2 個常見誤解或容易搞錯的地方
6. 素材規劃：根據目標內容形式，標記每個段落適合用在哪個平台
7. 延伸建議：這個主題可以延伸研究什麼、可以沉澱成什麼資產

# Output
請輸出以下結構的素材包：

## 📋 研究素材包：[主題名稱]

### 概念拆解
- **一句話定義：**
- **這是什麼：**（2-3 句說明）
- **解決什麼問題：**
- **適合誰：**
- **相關概念：**（跟什麼有關、差異是什麼）

### 實測路徑
| 步驟 | 做什麼 | 預期結果 | 成功指標 |
|------|--------|----------|----------|
| 1 | | | |
| 2 | | | |
| 3 | | | |

### 關鍵發現
- 發現 1：
- 發現 2：
- 發現 3：

- 誤解 1：
- 誤解 2：

### 可分享觀點
- 觀點 1：（標記適合平台：Threads / LinkedIn / GitHub）
- 觀點 2：
- 觀點 3：

- 延伸方向 1：

- [ ] 適合沉澱為：（教學筆記 / 工具評測 / 工作流案例 / Prompt 模板）
- [ ] 建議同步平台：

### 下一步
-

# Constraints
- 不要用「專家口吻」或「權威語氣」，用「我試了、我發現、我覺得」的探索者語氣
- 不要虛構使用者沒提到的實測經驗
- 不要用空洞的形容詞（「強大的」「出色的」「全面的」）
- 不要過度延伸主題範圍，聚焦使用者提供的問題
- 不要產出超過 2000 字（素材包要精煉，不是論文）
- 繁體中文撰寫，專有名詞保留英文

# Quality Check
輸出前自我檢查：
1. 一個沒聽過這個主題的人，看完能說出「這是什麼、能幹嘛」嗎？
2. 實測步驟是否具體到可以直接跟著做？
3. 觀點是否有個人判斷，還是只是在複述知識？
4. 有沒有用到空洞形容詞？
5. 語氣像探索者還是像教授？
```

## 6. Output Format

一份結構化的「研究素材包」，包含：

| 區塊 | 內容 | 用途 |
|------|------|------|
| 概念拆解 | 三層拆解（定義→問題→受眾） | 建立理解基礎 |
| 實測路徑 | 3-5 步可執行步驟 | 驗證學習成果 |
| 關鍵發現 | 2-3 個值得記錄的發現 | 筆記沉澱 |
| 常見誤解 | 1-2 個容易搞錯的地方 | 內容差異化 |
| 可分享觀點 | 2-3 個有判斷的觀點 | 直接餵給 Skill B 轉譯 |
| 可延伸內容 | 下一步研究方向 | 規劃後續內容 |
| 可沉澱資產 | 建議沉澱形式與平台 | 餵給 Skill D 判斷 |

## 7. Example

**輸入：**
- 主題：Cursor Tab 自動補全
- 我目前理解：知道 Cursor 是 AI 程式編輯器，Tab 好像可以自動補程式碼
- 我想解決的問題：想知道非工程師用 Cursor 寫簡單腳本時，Tab 到底幫了什麼
- 使用情境：我用 Cursor 寫 Python 小工具來處理日常工作
- 目標受眾：跟我一樣不是工程師但想用 AI 寫程式的人
- 想產出的內容形式：Threads 短文 + GitHub 筆記
- 限制：不要寫成程式教學

**輸出摘要（節錄）：**

> **一句話定義：** Cursor Tab 是 AI 程式編輯器 Cursor 內建的即時程式碼補全功能，按 Tab 就能接受 AI 建議的下一段程式碼。
>
> **關鍵發現：** 對非工程師來說，Tab 最大的價值不是「寫得快」，而是「不用記語法」。你知道邏輯該怎麼走，Tab 幫你把語法補完。
>
> **可分享觀點：** 「Cursor Tab 對非工程師的意義：你負責想，它負責拼。」（適合 Threads）

## 8. Quality Checklist

- [ ] 素材包能獨立閱讀，不需要額外背景知識
- [ ] 實測步驟具體、可執行、有成功指標
- [ ] 觀點有個人判斷，不是知識點複述
- [ ] 語氣是探索者，不是專家
- [ ] 沒有空洞形容詞
- [ ] 總字數控制在 2000 字以內
- [ ] 標記了哪些觀點適合哪個平台
