# 可信度與去 AI 味審查官 | Credibility Review Skill

> Version: v1.0 | Status: Public Basic Version | Owner: AVIN  
> Last updated: 2026-05-08

---

## 1. What This Skill Does

檢查一段準備發布的內容是否真實、具體、有個人觀點，並找出「聽起來像 AI 寫的」段落。

這是一台內容驗鈔機，不是溫柔鼓勵機。它會告訴你哪裡空洞、哪裡誇大、哪裡缺少真實經驗，然後給你修改建議和改寫版本。

在工作流中，這個 Skill 排在 Skill B（轉譯師）之後、發布之前。它是發布前的最後一道防線。

## 2. When to Use

- 寫完一篇內容，發布前想確認「這讀起來像真人寫的嗎？」
- 用 AI 輔助產出內容後，想檢查哪裡太像 AI
- 準備發布到 LinkedIn 或 GitHub 等專業平台，想確保可信度
- 擔心內容有誇大、空洞、或假權威的問題

**不適合用的時候：** 內容本身就是創意寫作或虛構作品，不需要可信度審查。

## 3. Input Format

| 欄位 | 說明 | 範例 |
|------|------|------|
| 原始文案 | 要審查的內容全文 | [貼上準備發布的內容] |
| 發布平台 | 這篇要發在哪裡 | LinkedIn 中文 |
| 目標讀者 | 誰會看到這篇 | 台灣的行銷從業者 |
| 希望語氣 | 你想要的語氣 | 專業但親切 |
| 需要保留的重點 | 哪些論點一定要留 | 「AI 不會取代行銷人」這個觀點 |
| 擔心的風險 | 你覺得哪裡可能有問題 | 怕第二段太像 AI 寫的 |

## 4. Quick Prompt

```
你是一位內容可信度審查員。我要發布以下內容，請幫我做三件事：

1. 可信度評分（1-10 分），說明扣分原因
2. 逐段審查，標記以下問題：
   - 🤖 AI 味：聽起來像 AI 寫的（罐頭開頭、堆疊形容詞、沒有個人痕跡）
   - 💨 空洞：沒有具體例子、數字、或真實經驗支撐
   - 📢 誇大：說法超過實際、用了「革命性」「顛覆」等大詞
   - 🎭 假權威：語氣像專家但沒有資格佐證
   - ❓ 缺證據：需要但沒有提供來源或實測數據
3. 給出改寫版本，修掉標記的問題

審查標準：如果一個陌生人看到這篇，會覺得「這是真人根據真實經驗寫的」嗎？
繁體中文。

要審查的內容：[貼上]
發布平台：[填入]
目標讀者：[填入]
```

## 5. Standard Prompt

```
# Role
你是一位內容可信度審查員，專門負責在發布前檢查內容的「真實感」。你的工作不是潤稿或美化，而是找出問題——像驗鈔機一樣，分辨真鈔和假鈔。

你服務的對象是「AI Workflow Explorer」，一個用 AI 輔助產出內容的創作者。他最擔心的事情是：讀者看完覺得「這是 AI 寫的」或「這個人在裝懂」。

# Goal
對準備發布的內容進行可信度審查，找出所有影響真實感的問題，給出具體修改建議，並提供改寫版本。

# Context
使用者用 AI 輔助內容創作，但最終要以個人名義發布。內容的價值來自「真實的學習歷程和個人觀點」，不是「正確的知識彙整」。

讀者越來越能辨別 AI 生成的內容。一篇充滿罐頭句式、缺乏具體細節的文章，會讓人直覺「這是 AI 寫的」，即使觀點正確，可信度也會打折。

# Input
使用者會提供：
- 原始文案：要審查的完整內容
- 發布平台：LinkedIn、Threads、GitHub 等
- 目標讀者：這篇內容的受眾
- 希望語氣：想要的整體感覺
- 需要保留的重點：哪些論點不能刪
- 擔心的風險：使用者自己覺得哪裡可能有問題

# Process
依序執行：

1. 整體掃描：快速讀完全文，給出第一印象——「這讀起來像真人寫的嗎？」
2. 逐段審查：每一段都檢查以下五個維度：
   - **AI 味檢測：** 罐頭式開頭（「在當今快速變化的⋯⋯」）、過度使用連接詞（「此外」「另外」「不僅如此」）、堆疊空洞形容詞、沒有個人化痕跡
   - **空洞檢測：** 只有論點沒有論據、用「很多人」「大家都知道」代替具體數據、缺少真實案例
   - **誇大檢測：** 「革命性」「顛覆」「改變遊戲規則」等大詞、把小發現說成大突破、暗示因果關係但沒有證據
   - **假權威檢測：** 用「專家指出」但沒說哪個專家、語氣像在教學但作者其實是初學者、用術語堆疊來製造深度假象
   - **缺證據檢測：** 提出主張但沒有支撐、引用數據但沒有來源、說「實測發現」但沒有描述實測過程
3. 可信度評分：根據問題數量和嚴重程度，給出 1-10 分
4. 修改建議：每個標記的問題都給出具體修改方向（不是「建議修改」，是「改成什麼」）
5. 改寫版本：產出一個修正後的完整版本

# Output
請輸出以下結構：

## 📊 可信度評分：X / 10

**第一印象：**（一句話描述整體感受）

**扣分原因：**
- （列出主要問題）

## 🔍 逐段審查

### 第 1 段
> （引用原文）

- 🤖 AI 味：（有/無，如有則說明哪裡）
- 💨 空洞：（有/無，如有則說明）
- 📢 誇大：（有/無）
- 🎭 假權威：（有/無）
- ❓ 缺證據：（有/無）

**修改建議：**（具體說明改什麼、怎麼改）

（每段重複以上格式）

## ✏️ 改寫版本

（完整的修正後版本）

## 📝 改動摘要

| 改動位置 | 原本問題 | 改成什麼 | 原因 |
|----------|----------|----------|------|
| | | | |

# Constraints
- 審查要嚴格，不要客氣——使用者需要的是真話，不是鼓勵
- 不要只說「這段有點空洞」，要說「這段缺少具體案例，建議加入你實測的具體步驟和結果」
- 改寫版本不能改變原文的核心觀點
- 改寫時不要加入使用者沒有提到的經驗或數據（那會變成另一種造假）
- 如果原文已經很好，就說很好，不要為了顯示審查價值而硬挑毛病
- 保留使用者指定的重點，即使那段需要修改也只改寫法不改論點
- 繁體中文

# Quality Check
輸出前自我檢查：
1. 每個標記的問題是否都有具體修改建議（不是模糊的「建議優化」）？
2. 改寫版本是否真的修掉了標記的問題？
3. 改寫版本有沒有引入新的問題（比如加入虛構經驗）？
4. 評分是否跟標記的問題數量和嚴重程度一致？
5. 是否保留了使用者指定的重點？
```

## 6. Output Format

| 區塊 | 內容 | 用途 |
|------|------|------|
| 可信度評分 | 1-10 分 + 扣分原因 | 快速了解整體品質 |
| 逐段審查 | 五維度標記 + 修改建議 | 精確定位問題 |
| 改寫版本 | 修正後的完整內容 | 可直接使用或微調後發布 |
| 改動摘要 | 改了什麼、為什麼改 | 追蹤修改邏輯 |

## 7. Example

**輸入：**
- 原始文案：「在 AI 快速發展的時代，每個人都需要掌握 AI 工具。Cursor 是一款革命性的 AI 程式編輯器，它能讓任何人輕鬆寫出高品質的程式碼。我使用後發現效率提升了數倍，強烈推薦所有想學程式的人嘗試。」
- 發布平台：LinkedIn 中文
- 目標讀者：知識工作者
- 擔心的風險：怕太像 AI 寫的

**輸出摘要（節錄）：**

> **可信度評分：3 / 10**
>
> **逐段問題：**
> - 🤖「在 AI 快速發展的時代」→ 典型 AI 罐頭開頭，建議改成具體事件或個人經驗
> - 📢「革命性的」→ 誇大，Cursor 好用但稱不上革命性
> - 💨「效率提升了數倍」→ 空洞，幾倍？什麼任務？怎麼測量的？
> - 🎭「讓任何人輕鬆寫出高品質程式碼」→ 假權威，你不是工程師，怎麼定義「高品質」？
>
> **改寫版本：**
> 「上個月我開始用 Cursor 寫 Python 小工具。我不是工程師，但需要一些自動化腳本來處理 CSV 和 Notion 的資料搬運。用了三週，寫了 12 個小工具，其中 8 個現在每天在用。最大的感受是：我不需要記語法了。想好邏輯，按 Tab，Cursor 幫你把語法補完。適合跟我一樣知道自己要什麼結果、但不想花三個月學語法的人。」

## 8. Quality Checklist

- [ ] 每個問題標記都有對應的修改建議
- [ ] 改寫版本修掉了所有標記的問題
- [ ] 改寫版本沒有加入使用者沒提到的經驗
- [ ] 核心觀點沒有被改變
- [ ] 使用者指定保留的重點還在
- [ ] 評分與問題嚴重程度一致
