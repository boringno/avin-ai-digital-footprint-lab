## 1. 快速理解

### 這套系統是什麼

一套把 AI 學習過程轉成跨平台數位資產的工作流系統。

你每次學一個 AI 工具、研究一個概念、實測一個 Prompt，這套系統幫你把這些過程整理成 Threads 貼文、LinkedIn 觀點文、GitHub 技術文件、履歷素材、個人網站案例。不是發完就忘，是每一次學習都累積成你的數位足跡。

核心元件：
- **Notion 同步中心**：追蹤所有內容的狀態和平台分布
- **四大 Skills**：四個可直接使用的 Prompt，負責拆解、轉譯、審查、資產判斷
- **平台同步規則**：什麼內容發到哪裡、用什麼格式

### 它不是什麼

- **不是自動發文機器。** 你還是要自己決定發什麼、什麼時候發。系統幫你整理和轉譯，不幫你按下發布鍵。
- **不是 AI 課程。** 不教你 AI 原理，教你怎麼把學 AI 的過程變成可見的資產。
- **不是要你變 AI 專家。** 定位是「AI Workflow Explorer」——邊學邊分享，不裝懂。

### 最小可行使用方式

只需要三樣東西就能開始跑：

1. **Notion 一個表**：建一個 Database，包含本文件 §4 列出的欄位
2. **四個 Prompt**：複製 §5 的 Quick Prompt 到你常用的 AI 工具（Claude、ChatGPT、Gemini 都行）
3. **手動同步**：每週花 30 分鐘，把高價值內容手動搬到其他平台

不需要任何付費工具、不需要寫程式、不需要設定自動化。先跑起來，之後再慢慢加。

## 2. 每天怎麼用

### 場景 A：今天學到一個 AI 工具

**觸發點：** 你試了一個新工具、看了一篇 AI 相關文章、或發現一個有趣的功能。

**做什麼：**
1. 打開 AI 助手，貼上 Skill A（AI 研究拆解師）的 Quick Prompt
2. 填入主題、你目前的理解、想解決的問題
3. 拿到素材包後，在 Notion 新增一筆紀錄
4. 把素材包的核心觀點填進「中文版摘要」欄位

**產出：** Notion 裡多了一筆有結構的學習紀錄，裡面有可以直接拿來發文的觀點。

### 場景 B：今天有一個 AI 想法

**觸發點：** 腦中閃過一個想法，但現在沒時間整理。

**做什麼：**
1. 在 Notion 同步中心新增一筆
2. 填入「主題名稱」和「備註」（隨手記幾句就好）
3. 「母內容狀態」設為「待拆解」

**產出：** 想法不會遺失，之後有空時用 Skill A 展開。

### 場景 C：今天產出了內容

**觸發點：** 你寫完一篇文章或發完一則貼文。

**做什麼：**
1. 打開 AI 助手，貼上 Skill D（數位足跡資產官）的 Quick Prompt
2. 填入內容主題、摘要、已發布平台
3. 根據建議，決定要不要同步到其他平台
4. 在 Notion 回填各平台的狀態和連結

**產出：** 你知道這份內容值多少、該放哪裡、下一步做什麼。

## 3. 每週怎麼用

### 每週回顧流程

每週挑一個固定時間（建議週日晚上或週一早上），花 30-60 分鐘做以下事情：

1. **打開 Notion 同步中心**，篩選「母內容狀態 = 完成」的項目
2. **檢查各平台狀態**：哪些只發了 Threads 但沒同步到 LinkedIn？哪些有 GitHub 潛力但還沒寫？
3. **挑選本週高價值內容**（用下面的三個問題判斷）
4. **執行同步**：用 Skill B 轉譯、用 Skill C 審查、然後發布
5. **回填連結**：發布後把各平台的連結貼回 Notion

### 怎麼判斷「高價值內容」

問自己三個問題：

1. **三個月後還有人想看嗎？** 如果是（長青內容），優先同步到 GitHub 和個人網站。
2. **能不能寫進履歷？** 如果能展示某個能力或成果，同步到履歷網站。
3. **有沒有獨特觀點？** 如果只有你有這個角度或經驗，優先同步到 LinkedIn。

三題都是「否」的內容，留在原平台就好，不需要花力氣同步。

### 同步決策矩陣

| 內容特性 | LinkedIn | GitHub | 履歷網站 | 個人網站 |
|----------|----------|--------|----------|----------|
| 有個人觀點和判斷 | ✅ 優先 | | | |
| 有技術細節或步驟 | | ✅ 優先 | | |
| 能展示能力或成果 | ✅ | | ✅ 優先 | |
| 有完整案例故事 | | | | ✅ 優先 |
| 長青、不會過時 | ✅ | ✅ | ✅ | ✅ |
| 時效性強、會過時 | | | | |

## 5. 四大核心 Skills 怎麼用

以下是每個 Skill 的精簡介紹和 Quick Prompt。完整版 Standard Prompt 請到 [`02-prompts-and-skills/`](02-prompts-and-skills/) 資料夾查看。

### Skill A：AI 研究拆解師

📄 完整版：[ai-research-breakdown-skill.md](02-prompts-and-skills/ai-research-breakdown-skill.md)

**用途：** 把一個 AI 學習主題拆成可理解、可實測、可分享的素材包。

**什麼時候用：** 學到新工具、看到新概念、想整理學習心得的時候。

**輸入：** 主題 + 你目前的理解 + 想解決的問題

**輸出：** 概念拆解、實測步驟、關鍵發現、可分享觀點、可沉澱資產

**Quick Prompt：**

```
你是一位 AI 工具探索者的研究助手。我會給你一個 AI 學習主題，請幫我拆解成以下結構：

1. 這是什麼（一句話說清楚）
2. 解決什麼問題（為什麼有人需要這個）
3. 適合誰用（什麼人、什麼場景）
4. 怎麼開始試（最小可行的實測步驟，3-5 步）
5. 我的發現（根據我提供的理解，整理出 2-3 個值得分享的觀點）
6. 常見誤解（1-2 個大家容易搞錯的地方）
7. 可以延伸什麼（下一步可以研究什麼）

語氣要求：像一個有好奇心的人在跟朋友分享，不是教授在上課。用繁體中文。

我的主題是：[填入主題]
我目前的理解：[填入你知道的]
我想解決的問題：[填入目的]
```

**使用範例：**
> 主題：Claude Code Hooks  
> 理解：知道是 Claude Code 的擴充功能  
> 問題：想知道能不能自動化開發流程  
> → 產出素材包，裡面有 3 個可以直接發 Threads 的觀點

### Skill B：多平台內容轉譯師

📄 完整版：[multi-platform-content-skill.md](02-prompts-and-skills/multi-platform-content-skill.md)

**用途：** 把一份母內容轉換成 7 個平台的版本。

**什麼時候用：** 素材包整理好了，要分發到各平台的時候。

**輸入：** 母內容 + 核心觀點 + 目標平台

**輸出：** 各平台的內容版本（Threads / IG / LinkedIn / GitHub / 履歷網站 / 個人網站）

**Quick Prompt：**

```
你是一位跨平台內容轉譯助手。我會給你一份母內容和核心觀點，請轉換成以下平台版本：

1. Threads：150-300 字，口語、有鉤子開頭、一個觀點講透就好
2. IG 輪播：5-7 張，第 1 張是標題鉤子，中間每張一個重點，最後一張是行動呼籲
3. LinkedIn 中文：800-1200 字，專業敘事、有個人經驗、不裝權威
4. LinkedIn 英文：600-1000 字，同上但英文，語氣稍正式、適合國際讀者
5. GitHub：README 格式，含標題、摘要、重點、連結
6. 履歷網站：1-3 句 project bullet，強調能力和成果
7. 個人網站：Case study 格式，含背景、做法、結果、學到什麼

每個版本要能獨立閱讀。語氣像探索者在分享，不是專家在教學。繁體中文為主。

母內容：[貼上]
核心觀點：[填入]
目標平台：[選擇要產出的平台]
```

**使用範例：**
> 母內容：Cursor Tab 素材包  
> 核心觀點：你負責想，它負責拼  
> 目標平台：Threads + LinkedIn  
> → 產出 Threads 短文 + LinkedIn 觀點文草稿

### Skill C：可信度與去 AI 味審查官

📄 完整版：[credibility-review-skill.md](02-prompts-and-skills/credibility-review-skill.md)

**用途：** 檢查內容是否真實、具體、有個人觀點，並去除 AI 罐頭感。

**什麼時候用：** 內容寫完、準備發布之前。

**輸入：** 原始文案 + 發布平台 + 目標讀者

**輸出：** 可信度評分（1-10）+ 逐段審查 + 改寫版本

**Quick Prompt：**

```
你是一位內容可信度審查員。我要發布以下內容，請幫我做三件事：

1. 可信度評分（1-10 分），說明扣分原因
2. 逐段審查，標記以下問題：
   - 🤖 AI 味：聽起來像 AI 寫的
   - 💨 空洞：沒有具體例子或真實經驗
   - 📢 誇大：說法超過實際
   - 🎭 假權威：語氣像專家但沒有資格佐證
   - ❓ 缺證據：需要但沒有提供來源
3. 給出改寫版本，修掉標記的問題

審查標準：陌生人看到這篇，會覺得「這是真人根據真實經驗寫的」嗎？繁體中文。

要審查的內容：[貼上]
發布平台：[填入]
目標讀者：[填入]
```

**使用範例：**
> 審查一篇 LinkedIn 草稿  
> → 評分 6/10，標記 2 處 AI 味、1 處空洞  
> → 產出改寫版本，可信度提升到 8/10

### Skill D：數位足跡資產官

📄 完整版：[digital-asset-review-skill.md](02-prompts-and-skills/digital-asset-review-skill.md)

**用途：** 判斷一份內容應該沉澱成什麼長期資產，建議同步平台。

**什麼時候用：** 內容通過審查、準備發布或剛發布完的時候。

**輸入：** 內容主題 + 摘要 + 已發布平台 + 商業化可能

**輸出：** 資產分類、資產等級、平台同步建議、隱形資產、Notion 欄位建議、下一步

**Quick Prompt：**

```
你是一位數位足跡資產顧問。我會給你一份已完成的內容，請幫我判斷：

1. 資產分類：教學型 / 觀點型 / 工具評測型 / 案例型 / 模板型
2. 資產等級（深度 × 獨特性 × 時效性）：A 級 / B 級 / C 級
3. 建議同步平台：哪些適合、哪些不適合、為什麼
4. 隱形資產：能變成履歷 bullet？能用在提案？能商業化？
5. Notion 欄位建議：該怎麼填
6. 下一步：接下來做什麼

繁體中文。

內容主題：[填入]
內容摘要：[填入]
目前狀態：[填入]
已發布平台：[填入]
```

**使用範例：**
> 主題：Cursor Tab 實測  
> → B+ 級資產（獨特性高，非工程師視角少見）  
> → 優先同步 LinkedIn → GitHub → 履歷網站  
> → 隱形資產：可佐證「獨立建構 AI 工作流」能力

## 6. 平台同步規則

### Threads

| 項目 | 說明 |
|------|------|
| 適合什麼 | 短觀點、學習心得、工具速評、日常 AI 使用感想 |
| 格式 | 純文字，150-300 字，第一句要有鉤子 |
| 語氣 | 口語、輕鬆、像跟朋友聊天 |
| 頻率 | 每週 3-5 則 |
| 範例場景 | 今天試了一個新 AI 功能，想快速分享一個發現 |

### IG

| 項目 | 說明 |
|------|------|
| 適合什麼 | 教學型內容、步驟拆解、視覺化整理 |
| 格式 | 輪播 5-7 張，每張一個重點，文字精簡 |
| 語氣 | 清楚、有設計感、適合截圖分享 |
| 頻率 | 每週 1-2 則 |
| 範例場景 | 把一個 AI 工具的使用步驟做成輪播教學 |

### LinkedIn 中文

| 項目 | 說明 |
|------|------|
| 適合什麼 | 有觀點的專業分享、工具深度評測、工作流案例、職涯相關的 AI 觀察 |
| 格式 | 段落式，800-1200 字，有小標題 |
| 語氣 | 專業但不生硬，有個人經驗，不裝權威 |
| 頻率 | 每週 1-2 篇 |
| 範例場景 | 分享用 AI 改善工作流程的真實經驗和觀點 |

### LinkedIn 英文

| 項目 | 說明 |
|------|------|
| 適合什麼 | 同上，但面向國際讀者、想建立英文專業形象的內容 |
| 格式 | 段落式，600-1000 字 |
| 語氣 | 稍正式、國際化、可被搜尋 |
| 頻率 | 每月 2-4 篇（挑高價值內容翻譯） |
| 範例場景 | 把表現好的中文 LinkedIn 文章改寫成英文版 |

### GitHub

| 項目 | 說明 |
|------|------|
| 適合什麼 | Prompt 模板、工作流文件、工具評測紀錄、技術筆記、開源 Skill |
| 格式 | Markdown，README 風格，結構化 |
| 語氣 | 精確、技術友善、適合開發者閱讀 |
| 頻率 | 有新的可沉澱內容時就更新 |
| 範例場景 | 把實測好用的 Prompt 整理成 .md 檔放上 GitHub |

### 履歷網站

| 項目 | 說明 |
|------|------|
| 適合什麼 | 能展示能力或成果的內容，可量化的專案 |
| 格式 | Project bullet，1-3 句，動詞開頭 |
| 語氣 | 精煉、成果導向 |
| 頻率 | 有 A/B 級資產時同步 |
| 範例場景 | 用 AI 建了一套工作流系統，寫成履歷上的一個專案 |

### 個人網站

| 項目 | 說明 |
|------|------|
| 適合什麼 | 完整案例故事、深度教學、作品展示 |
| 格式 | Case study（背景→做法→結果→學到什麼），400-800 字 |
| 語氣 | 專業、有深度、適合願意花時間的讀者 |
| 頻率 | 每月 1-2 篇（只放 A 級資產） |
| 範例場景 | 把「從零建立 AI 數位足跡系統」寫成一個完整案例 |

## 7. 開源與私有邊界

### 可以公開

| 內容 | 原因 |
|------|------|
| 四大 Skills 的 Prompt（Quick + Standard） | 這是公開數位足跡的一部分，展示你的方法論 |
| 工作流說明和 SOP | 幫助別人複製你的方法，建立社群信任 |
| 工具評測和實測紀錄 | 真實經驗是你的差異化，公開不會損失價值 |
| Notion 欄位結構（不含實際資料） | 模板本身可公開，資料不行 |
| 本使用說明文件 | 展示系統化思維 |

### 半公開

| 內容 | 原因 |
|------|------|
| 個人案例（脫敏後） | 可以分享做法和結果，但要移除敏感細節 |
| 具體的內容產出範例 | 展示品質，但可能包含個人資訊需處理 |
| 自動化流程的架構圖 | 可以分享架構概念，但不需公開完整實作 |

### 保留私有

| 內容 | 原因 |
|------|------|
| 客戶相關的任何內容 | 保密義務 |
| 商業策略和變現計畫 | 競爭優勢 |

## 8. 免費自動化策略

### 核心原則

這套系統不是反自動化。自動化是目標，但我們用免費方式做。

付費自動化工具（n8n Cloud、Zapier、Make）月費 $20-100 美元，在你還沒驗證這套系統能穩定運作之前，不值得花這個錢。先用免費工具跑通流程，確認有效再考慮升級。

### 優先使用

| 工具 | 用途 | 費用 |
|------|------|------|
| GitHub Actions | 定時任務、自動生成、CI/CD | 免費（公開 repo） |
| Notion API | 讀寫 Notion Database、自動更新欄位 | 免費 |
| GitHub Issues | 追蹤任務、觸發自動化流程 | 免費 |
| GitHub Pages | 靜態網站託管、個人網站 | 免費 |
| Claude / ChatGPT | 手動跑 Skill Prompt | 依方案 |

### 不優先使用

| 工具 | 原因 |
|------|------|
| n8n Cloud 付費方案 | 先用免費替代方案驗證流程 |
| Zapier | 免費額度太少，付費門檻高 |
| Make (Integromat) | 同上 |

### 免費自動化範例

**範例 1：GitHub Actions 自動生成每週回顧**
- 每週日晚上自動跑一個 Action
- 讀取 Notion 同步中心的本週資料
- 產出一份 Markdown 格式的週報
- 自動 commit 到 GitHub repo

**範例 2：Notion API 自動更新狀態**
- 當 GitHub 有新 commit 時，透過 webhook 更新 Notion 的「GitHub 狀態」欄位
- 不需要手動回填

**範例 3：GitHub Issues 作為任務追蹤**
- 每次用 Skill D 判斷要同步的平台，自動建立 GitHub Issue
- Issue 標題：「同步 [主題] 到 [平台]」
- 完成後關閉 Issue

**範例 4：LinkedIn 草稿自動生成**
- GitHub Actions 定時讀取 Notion 中「LinkedIn 狀態 = 未開始」的項目
- 自動用 Skill B 的 Prompt 產出 LinkedIn 草稿
- 存成 Markdown 檔放在 `drafts/linkedin/` 資料夾
- **不自動發布**——LinkedIn 內容一定要人工審核後手動發

## 10. 版本更新規則

### 什麼情況觸發更新

| 觸發條件 | 更新範圍 | 版本變動 |
|----------|----------|----------|
| 新增或移除 Skill | §5 四大核心 Skills + 對應 Skill 檔案 | 次版本（如 v1.1 → v1.2） |
| 新增或移除同步平台 | §4 欄位說明 + §6 平台同步規則 | 次版本 |
| 欄位結構調整 | §4 欄位說明 | 次版本 |
| 整體架構重構 | 全文件 | 主版本（如 v1.x → v2.0） |
| 修正錯字或小調整 | 該段落 | 不改版本號 |

### 更新 SOP

1. 在使用說明的 header 更新日期和版本號
2. 修改對應章節內容
3. 如果是 Skill 變動，同步更新 `02-prompts-and-skills/` 資料夾的檔案
4. 在 GitHub commit message 中註明「docs: update guide to vX.X — [變更摘要]」
5. 如有需要，同步更新 Notion 同步中心的欄位結構
